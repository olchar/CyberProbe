📘 What is the Investigation Guide?

The Defender XDR Investigation Guide is a comprehensive reference manual and automation framework for investigating security incidents using Microsoft's security stack:

  • Microsoft Defender XDR - Extended Detection & Response
  • Microsoft Sentinel Data Lake - SIEM with KQL query interface
  • Microsoft Graph API - Identity and device data
  • MCP (Model Context Protocol) Servers - Programmatic access layer
🎯 Unique Feature: This guide serves dual purposes:
1. 📖 Reference Manual - Human-readable documentation for analysts learning investigation techniques
2. 🤖 Automation Instructions - Step-by-step workflows for GitHub Copilot and AI-assisted investigations

🚀 Quick Start Guide

For New Users (Manual Investigations)

Scenario: Investigating a Suspicious User Sign-in Alert

  1. Navigate to Section 3 (Investigation Types)
    Choose your scope: Standard (7 days), Quick (1 day), or Comprehensive (30 days)
  2. Go to Section 6 (Investigation Workflows)
    Find "User Investigation" workflow and copy the basic User Activity Timeline query
  3. Check Section 8 (Sample KQL Queries)
    • Use Query 2 (Anomalies from Detection System)
    • Use Query 3/3b/3c (Sign-ins by app/location/failures)
  4. If Geographic Anomaly Detected
    Go to Section 9 (Advanced Authentication Analysis) and follow the 6-step SessionId forensic tracing workflow
  5. Document Findings
    Use Section 17 (Investigation Report Template) to export standardized report
⏱️ Expected Time: 30-45 minutes for standard 7-day investigation

For AI-Assisted Investigations (Automation)

Scenario: Automating User Risk Investigation

  1. Tell Copilot:
    "Investigate user jane.doe@company.com for the last 7 days using the Investigation Guide workflow"
  2. Copilot Will Automatically:
    • Check Section 1 (Critical Workflow Rules)
    • Get User Object ID + Windows SID from Graph API
    • Run Query 2 (anomalies) to determine scope
    • If anomalies exist: Run Query 1 → 3d → 11
    • Execute Advanced Authentication Analysis if needed
    • Export JSON with all 20+ required fields
  3. Review Output:
    JSON file in temp/ directory with complete investigation data, IP enrichment, and risk assessment
⏱️ Expected Time: 2-5 minutes (automated execution)

🔍 Common Investigation Workflows

Workflow 1: Geographic Anomaly Investigation

Alert: User signed in from Nigeria, but normally works in Seattle.

  1. Section 8 → Query 2: Check if anomaly system flagged this
    Signinlogs_Anomalies_KQL_CL | where UserPrincipalName =~ 'user@company.com' | where DetectedDateTime between (datetime(2026-01-01) .. datetime(2026-01-09))
  2. Section 9 → Step 1: Get SessionId from Nigeria IP
  3. Section 9 → Step 2: Trace complete authentication chain
  4. Section 9 → Step 3: Find interactive MFA event (first in chain)
  5. Section 9 → Step 6: Document risk assessment
🔑 Key Question: Was the initial MFA from Seattle (corporate VPN) or Nigeria (unknown IP)?

• Seattle VPN first → Legitimate user traveling → No action
• Nigeria IP first → Credential compromise → Force password reset

⏱️ Time: 15-20 minutes

Workflow 2: Phishing Campaign Response

Alert: 5 users clicked link in suspected phishing email.

  1. Section 12 → Phishing Playbook: Follow step-by-step guide
  2. Section 8 → Query 5: Check Office 365 activity for email actions
  3. Section 8 → Query 4: Check Azure AD audit logs for credential changes
  4. Section 8 → Query 2: Check for post-compromise anomalies
  5. Section 11 → IP Enrichment: Enrich suspicious IPs from email headers
  6. Section 17 → Report Template: Document incident

⏱️ Time: 1-2 hours for 5 users

Workflow 3: Automated Daily User Risk Check

Use Case: Nightly automated scan for high-risk user activity.

  1. Configure GitHub Copilot with MCP servers (see Section 10)
  2. Create scheduled task to invoke investigations
  3. Copilot follows Section 2 automation workflow automatically
  4. Review morning report - Parse JSON for high-risk findings

⏱️ Time: 2-5 minutes per user (automated)

💡 Tips for Effective Use

For Manual Investigations

1. Start with Sample Queries

Always use Section 8 queries first - they handle edge cases like missing fields, dynamic JSON, and schema differences.

Example: LocationDetails requires parse_json() - sample queries do this correctly

2. Use Correct Date Ranges

Real-time searches: Add +2 days to end date
Historical searches: Add +1 day to end date

Today is Jan 7, "last 7 days" = datetime(2026-01-01) to datetime(2026-01-09)

3. Check Troubleshooting First

Section 16 has solutions for common errors:

  • SemanticError → Field not in schema
  • Query timeout → Reduce date range
  • Graph 404 → Verify UPN spelling

4. Document Empty Results

"No anomalies detected" is valuable information - shows baseline behavior and is required for comprehensive reports.

5. Get User IDs for Incidents

Query 6 requires both User Object ID AND Windows SID from Graph API for complete incident correlation.

/v1.0/users/<UPN>?$select=id,onPremisesSecurityIdentifier

For Automation

1. Always Provide Current Date

Bad: "Investigate user for last 7 days"

Good: "Today is Jan 7, 2026. Investigate user for last 7 days"

2. Specify Investigation Type

  • "Standard investigation (7 days)" → Default
  • "Quick investigation (1 day)" → Fast turnaround
  • "Comprehensive (30 days)" → Deep forensics

3. Check Existing JSON First

Copilot should search temp/ directory before running queries to avoid duplicate API calls.

4. Request Timing Information

"Track and report timing after each major step" helps identify slow queries or API bottlenecks.

5. Validate All JSON Fields

  • department → "Unknown" if null
  • anomalies → Empty array [] if none
  • ipEnrichment → Mandatory (4 sources)

📊 Understanding Investigation Outputs

JSON Investigation Export Structure

Location: temp/investigation_<UPN>_<DATE>.json

{ "user_principal_name": "jane.doe@company.com", "investigation_start": "2026-01-01T00:00:00Z", "investigation_end": "2026-01-09T23:59:59Z", "anomalies": [ { "detected_datetime": "2026-01-05T14:32:00Z", "anomaly_type": "NewCountryIP", "value": "102.89.3.15", "severity": "High", "country": "NG" } ], "signin_ip_counts": [ { "ip_address": "102.89.3.15", "signin_count": 12, "last_auth_result_detail": "MFA requirement satisfied by claim in the token" } ], "ip_enrichment": [ { "ip": "102.89.3.15", "is_vpn": false, "abuse_confidence_score": 72, "threat_description": "Credential stuffing attacks", "city": "Lagos", "country": "NG" } ], "risk_level": "High", "recommendation": "Force password reset + revoke all sessions" }

How to Interpret Results

Field What It Means Action Required
anomalies.length > 0 Detection system flagged unusual activity Investigate with Section 9 (SessionId tracing)
abuse_confidence_score > 50 High-risk IP with confirmed malicious activity Block IP, force password reset
is_vpn = true IP is known VPN/proxy Check if corporate VPN (org IP ranges)
last_auth_result_detail = "Token" No interactive MFA, token refresh only Check wider date range for interactive sign-in
risk_level = "High" Combined risk assessment indicates compromise Follow recommendation field (force reset, revoke sessions)

🎯 Advanced Techniques

SessionId-Based Forensic Tracing (Section 9)

Why This Matters:
• Standard sign-in analysis shows WHERE user signed in
• SessionId tracing shows WHEN and HOW user FIRST authenticated
• Distinguishes legitimate travel from stolen credentials

Real-World Case Study

Initial Alert: User signs in from Lagos, Nigeria (flagged as anomaly)

Standard Analysis: "Suspicious foreign IP, recommend password reset"

⬇️

SessionId Tracing: Initial MFA was from Seattle corporate VPN 2 hours earlier

⬇️

Conclusion: User traveling for work, authenticated on trusted network first

Action: No response needed, document baseline expansion

Without SessionId tracing: False positive, disrupted legitimate user
With SessionId tracing: Accurate risk assessment, no user impact

Parallel Query Execution (Section 15 → Rule 5)

Query Type Queries Execution
Independent (Parallel) Query 2, 3, 3b, 4, 5, 10 Run simultaneously
Dependent (Sequential) Query 1 → 3d → 11 Run in order
⏱️ Time Savings: Parallel execution reduces 6-query workflow from ~3 minutes to ~30 seconds
